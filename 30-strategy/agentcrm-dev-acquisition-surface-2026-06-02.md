# agentcrm.dev Acquisition Surface

## Business outcome

Use `agentcrm.dev` to reduce CAC for developer and agent-builder audiences without creating a second product strategy or second roadmap.

## Decision

`agentcrm.dev` should be used as a **developer-facing acquisition surface**, not as a separate standalone product bet.

Useful roles:

- install/docs surface for MCP and API integrations
- SEO surface for developer-intent searches around agent CRM and customer memory
- microsite for examples, starter repos, prompts, and reference architectures
- low-friction path into UpSight

Not useful roles:

- second company story
- separate open-source CRM roadmap
- split brand with separate monetization logic

## Why

- We already have working MCP integrations in Claude, ChatGPT, and OpenClaw.
- A developer-facing surface can package something real instead of hand-wavy future vision.
- "CRM for agents" is strategically weak if taken literally; customer memory and evidence are stronger.

## Positioning

Prefer:

- **Customer memory for AI agents**
- **Customer intelligence your agents can reason over**
- **MCP + API layer for evidence-backed customer context**

Avoid:

- "open-source CRM for agents"
- "the new AI CRM"

## Relationship to UpSight

- `agentcrm.dev` = developer acquisition and packaging layer
- UpSight = monetized product and human workspace

The domain should feed into:

- API keys
- MCP install docs
- examples
- "open in UpSight" moments
- paid workspace expansion

## Recommended site sections

- homepage
- install
- use cases
- examples
- why UpSight
- get API key

## Guardrails

- no separate feature commitments
- no separate pricing strategy
- no standalone CRM buildout

## Success criteria

`agentcrm.dev` is worth it if it produces:

- developer signups
- MCP installs
- API key generation
- traffic from agent-builder keywords
- one agent-builder design partner or customer
